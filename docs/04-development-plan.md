# 개발 플랜

## 전체 일정 (12주, 사이드 업무 기준)

본 프로젝트는 **전담이 아닌 사이드 업무**를 전제로 한다.
- 권장 투입: 주 6~10시간 (평일 1~2회 + 주간 점검)
- 일정 산정 방식: "작업일"이 아니라 "주차" 기준
- 원칙: **무리한 병렬 개발보다 단계별 데모/검증 우선**

```
1-2주차: 착수 준비 + 인프라/보안 검증
3-4주차: 1단계(담당자 안내 봇) 구현/데모
5-8주차: 2단계(규정 안내 봇) 데이터·로직 고도화
9-10주차: 통합 테스트/튜닝 + 운영체계 확정
11주차: 2단계 데모 오픈
12주차: 전사 오픈 + 운영 전환
```

---

## 0. 착수 준비 (W1)

### 목표
권한/보안/운영 주체를 먼저 확정해 이후 개발 리스크를 줄인다.

### 작업 항목

| # | 작업 | 담당 | 산출물 |
|---|------|------|--------|
| 0-1 | 네이버웍스 봇 개발·운영 가능 여부 확인 (보안/인프라) | 본인 + IT/보안 | 가능/불가 판정 |
| 0-2 | 보고/의사결정 라인 확정 | 본인 + 본부장 | 의사결정 체계 |
| 0-3 | 1단계/2단계 범위 고정 (스코프 동결) | 본인 | 범위 문서 |
| 0-4 | 주간 운영 리듬 확정 (회의/리뷰 슬롯) | 본인 + 실무자 | 주간 캘린더 |

### 완료 기준
- 인프라·권한 Blocker 없음
- 1/2단계 범위 합의 완료
- 주간 리뷰 슬롯 확보

## 1. 인프라 및 베이스라인 측정 (W2)

### 목표
에코봇 수준의 연결 검증 + 성과 측정 기준선 확보.

### 작업 항목

| # | 작업 | 담당 | 산출물 |
|---|------|------|--------|
| 1-1 | Bot 생성/Service Account/Key 발급 | 본인(관리자 협조) | Bot ID, SA, Key |
| 1-2 | n8n Webhook 연결 + 에코 응답 테스트 | 본인 | 송수신 확인 |
| 1-3 | Gemini API 단독 호출 테스트 | 본인 | 호출 성공 로그 |
| 1-4 | 도입 전 베이스라인 수집 (문의량/응대시간/대기시간) | 본인 + 경영지원 | 기준선 데이터 |

### 완료 기준
- 메시지 수신/응답 end-to-end 성공
- 베이스라인 지표 수집 완료

## 2. 1단계: 담당자 안내 봇 (W3-W4)

### 목표
FAQ 우선 + 에스컬레이션 중심의 **1단계 데모 오픈**.

### 작업 항목

| # | 작업 | 산출물 |
|---|------|--------|
| 2-1 | 부서별 반복 질문 수집/정리 | 질문 리스트 |
| 2-2 | FAQ 데이터 초안 구축 (핵심 60~100건) | FAQ 시트 |
| 2-3 | 담당자 매핑 시트 구축 (category/subCategory) | 담당자 시트 |
| 2-4 | n8n 담당자 찾기 로직 구현 | Tier3 안내 동작 |
| 2-5 | 기본 로깅(문의/카테고리/에스컬레이션) 구현 | 운영로그 |
| 2-6 | 1단계 데모 오픈 (소수 사용자) | 데모 결과 |

### Gate A (W4 종료)
- FAQ 질문에 Tier1 즉시 응답
- 답변 불가 시 담당자 안내 정확 동작
- 치명 오류 없음 (데모 운영 가능)

## 3. 1단계 안정화 + 2단계 설계 (W5-W6)

### 목표
1단계 피드백 반영 및 2단계(규정 안내)용 데이터 구조 확정.

### 작업 항목

| # | 작업 | 산출물 |
|---|------|--------|
| 3-1 | 1단계 데모 피드백 반영 (FAQ/담당자 보강) | 수정 내역 |
| 3-2 | 2단계 시나리오·데이터 구조 상세 설계 | 설계서 |
| 3-3 | 데이터 표준화 가이드 작성 (FAQ/규정 입력 규칙) | 운영 가이드 |
| 3-4 | FAQ_키워드인덱스 구조 및 운영 규칙 확정 | 인덱스 시트 |

### 완료 기준
- 2단계 구현 스펙 확정
- 실무자 입력 가이드 공유 완료

## 4. 2단계: 규정 안내 로직 구현 (W7-W8)

### 목표
규정 기반 응답(Tier2) + 세션 기반 멀티턴을 실사용 가능 수준으로 구현.

### 작업 항목

| # | 작업 | 산출물 |
|---|------|--------|
| 4-1 | 규정 데이터 정제 (Sheet/Docs/Confluence 소스 정리) | 규정 시트 |
| 4-2 | 분기 처리 및 멀티턴 로직 구현 | 워크플로우 |
| 4-3 | Gemini 의도 분류/응답 생성 프롬프트 반영 | 프롬프트 v2 |
| 4-4 | 하이브리드 조회(FAQ 인덱스 + 규정 조회) 구현 | 통합 흐름 |
| 4-5 | 2단계 대시보드 초안 구축 | 대시보드 초안 |

### 완료 기준
- FAQ 미매칭 질문에 규정 기반 응답 생성
- 동일 세션 후속 질문 문맥 유지 확인

## 5. 통합 테스트/튜닝 + 운영 준비 (W9-W10)

### 목표
2단계 데모 직전 품질 안정화 및 운영 거버넌스 확정.

### 작업 항목

| # | 작업 | 산출물 |
|---|------|--------|
| 5-1 | 카테고리별 테스트 시나리오 실행 | 테스트 리포트 |
| 5-2 | 멀티턴 시나리오 집중 점검 | 멀티턴 리포트 |
| 5-3 | FAQ/규정 갭 보강 | 데이터 업데이트 |
| 5-4 | 권한/시트 보호/장애 대응 절차 확정 | 운영 체크리스트 |
| 5-5 | KPI/ROI 대시보드 수식 검증 | 대시보드 v1 |

### Gate B (W10 종료)
- FAQ/규정/에스컬레이션 3-Tier 정상 동작
- 멀티턴 핵심 시나리오 통과
- 운영 권한/장애 대응 문서 확정

### 작업 항목

| 6-4 | 전사 오픈 + 월간 운영 루틴 이관 | 운영 전환 완료 |

- 주간/월간 운영 루틴 실행 가능
- KPI/ROI 보고 체계 정상 가동

## 사이드 업무 전제의 운영 원칙

- **주간 최소 단위 산출물 1개**를 원칙으로 한다 (작게라도 매주 진척 확보).
- 평일 실무가 바쁠 경우, 데모/게이트 일정은 고정하고 세부 작업만 조정한다.
- 신규 요구사항은 해당 단계 종료 전에는 원칙적으로 backlog로 이관한다.
- 일정 지연 시 우선순위: ①장애/보안 ②에스컬레이션 정확도 ③FAQ 커버리지 ④응답 품질 고도화.

## 주요 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 관리자 권한/보안 승인 지연 | 초기 일정 지연 | W1에 선확인, Blocker 즉시 보고 |
| FAQ/규정 데이터 입력 지연 | 정확도 저하 | 핵심 질문 우선(상위 빈도 60%) |
| 사이드 업무로 인한 투입 변동 | 주차 지연 | 게이트 중심 관리, 범위 동결 |
| 멀티턴 오작동 | 사용자 신뢰 저하 | 세션 타임아웃·후속질문 케이스 집중 테스트 |
| 운영 담당자 부재 | 오픈 후 운영 리스크 | 주/월간 루틴 담당자 백업 지정 |
